## Findings, in order of importance

### 1. The bottleneck shifted from "model" to "validator"

In the answer-thinking-on run, the 4 remaining concise-summary fallbacks (A04, A10, A26, A30) all failed with the same validator error: `unsupported_numeric_or_ref_claim`. The LLM produced rich multi-claim answers; the validator's `unit_aware_strict` semantic-claim guard rejected them; the no-think repair pass could not fix them (`repair_success = 0/4`); the deterministic fallback template took over.

This is the same `answer_claim_guard_mode = unit_aware_strict` knob the overnight report flagged. With Qwen's voice (especially with thinking on) the guard over-flags. Concrete next experiment: switch to `current_strict` (the previous default) and re-run; if cleaner, that becomes the new default. If `current_strict` overshoots and loses faithfulness, the third tier is enabling thinking on the repair call too (currently off to bound latency).

### 2. Sampling parameters mattered, but not as much as the synthesis-stage thinking flag

- llama.cpp defaults → Unsloth non-thinking preset: pass_rate 0.933 → 1.000, lex_mm not directly comparable, validator-trigger rate +1, but the headline jumped because A04 was de-temporalized in parallel. Hard to separate the two effects in a single run.
- Unsloth non-thinking → answer-thinking-on: lex_mm 7 → 5, validator-trigger rate 11 → 5, fallback rate 8 → 4. **This** is where the real quality move happened. Thinking on the synthesis stage gives Qwen room to actually read its evidence; without it, the model produces shallower answers that more often miss the question's keyword and trip the claim guard at the same time.

### 3. Five cases freed by answer-thinking-on, three new cases surfaced

Freed (in the prior run, lex_mm; now clean): **A08, A09, A21, A23, A25**. These were synthesis-voice problems — answers that didn't echo the must_include keyword even though their citations were correct.

Newly surfaced: **A22, A26, A30**. A22 is a false positive (DOGV doc just doesn't use the word "rural"; answer is substantively correct). A26 and A30 are stuck in the validator-fallback path described in (1).

Persisting from both runs: **A03** (ambiguous query — `2025/XXXX` placeholder reference, system gracefully handles but doesn't echo the keyword) and **A04** (concise-summary fallback). A04 is the most informative case: 8 genuinely relevant citations spanning both movilidad (SICUE, Becas Santander, PMR taxis) and transporte (school, university), but the answer stays in the fallback voice. Validator killed the first-pass answer, repair failed.

### 4. Retrieval looks fine on this set, with one weak case

Across all 7 problem cases inspected against ground-truth document titles, **no instance** of "wrong document retrieved" was found. The one weak case is **A30**, which asked for both vivienda and transporte but retrieval pulled 8 vivienda docs and 0 transporte docs. Multi-topic queries are weakly served by single-vector retrieval; not blocking the gate, but worth filing as a separate concern. The retrieval gate (`data/eval_reports/20260526T124516Z.json`) has not been re-run since the migration — re-running it would close the loop on retrieval quality post-migration.

### 5. Per-stage latency profile (answer-thinking-on run, 30 cases)

```
stage           p50      p95      change vs no-think Unsloth run
```

Per-question wall avg ~180s, p95 ~230s. The answer-stage cost is the entire delta. UX-wise this is too slow for interactive use — for shipping, we either accept the wall (overnight batch / async pattern) or keep the synthesis no-think and fix the validator instead.
