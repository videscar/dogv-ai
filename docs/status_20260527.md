## The Qwen latency problem and how it was fixed

### Symptom

End-to-end `/ask` against the Qwen3-27B chat server took 4-19 minutes per question. Per-stage timing on a representative DOGV question (case A01 from `eval_answer_demo_v1.json`):

```
WITH Qwen3 reasoning (baseline):
  intent     51-213s
  retrieve   30s
  rerank     45-120s  (often hit RERANK_TIMEOUT, fell back to base RRF)
  read       112-208s
  answer    >100s   (often timed out)
  TOTAL      ~10-19 min per /ask
```

For comparison, the gpt-oss-20b baseline (Feb 26 demo) was ~17s per /ask.

### Diagnosis

- Qwen3 is a reasoning model. With `--reasoning-format auto` on the chat server, llama.cpp routes `<think>...</think>` content into a separate `reasoning_content` field, but those tokens **still count against `max_tokens`** during generation.
- Each LLM call burned 3000-5000 reasoning tokens before producing a 100-500 token answer.
- The pipeline has 4 sequential LLM calls per /ask (intent → rerank → read → answer), so reasoning cost multiplies.
- Bumping `LLM_MAX_TOKENS` from 4096 → 8192 (earlier WIP commit) made it worse — Qwen filled the bigger budget.

### Probe

Three single-shot tests against `/v1/chat/completions` on a trivial classifier prompt:

```
A. baseline (thinking on):          5s,  666 reasoning tokens, 192 completion
B. /no_think suffix in user msg:    9s, 1012 reasoning tokens (ignored), 330 completion
C. chat_template_kwargs:
     {enable_thinking: false}:      1s,    0 reasoning tokens,  21 completion  ← WIN
```

`/no_think` as a prompt suffix is a Qwen-team convention that the model can choose to honor or ignore. The llama.cpp jinja template's `enable_thinking=false` parameter forces it off at the template level and is reliable.

### Fix

`api/llm.py` gets a new `enable_thinking: bool = True` parameter on `chat()` and `chat_json()`. When False, the request payload includes `chat_template_kwargs: {enable_thinking: false}`, which llama.cpp's Qwen3 jinja template uses to omit the `<think>` block entirely.

All deterministic-JSON call sites in the pipeline now pass `enable_thinking=False`:

```
api/intent.py          ✓ (both analyze_intent and analyze_intent_and_expand)
api/rerank.py          ✓
api/query_expansion.py ✓
api/classify.py        ✓
api/reader.py          ✓
api/answer.py          ✓ (via _chat_json_with_retry, default False)
```

`scripts/audit_eval_set.py` left alone — it is an offline gold-set auditing tool, not the runtime pipeline, and benefits from reasoning when judging ambiguous cases.

### Recommendation on the failing A04

Two options for restoring a green gate, ordered by effort:

1. **De-temporalize A04 in the eval set** (`data/eval_answer_demo_v1.json`). Re-phrase to a question that should be true regardless of "today" — e.g. "¿Qué subvenciones de movilidad o transporte para estudiantes universitarios se publicaron en 2025?" or scope to a fixed month. Keeps the test critical without coupling it to the run date. Same fix should be applied to A13.
2. **Backfill the most recent few days of DOGV** so "esta semana" has data. Re-run `python scripts/maintain_indices.py --daily` (or `--bootstrap` for a wider window). Then A04 may pass again. But this couples the gate to ongoing ingest freshness — the underlying eval-set fragility remains.

Option 1 is the proper fix for a ship-ready test gate. Option 2 is a workaround.

## Recommendations and follow-ups

1. **De-temporalize A04 + A13** in `data/eval_answer_demo_v1.json`. They are the only failures and both fail for the same temporal reason. Restoring a green gate is a 5-minute eval-set edit. Suggested rewrites in the [Failure analysis](#failure-analysis) section. Re-running just those two cases after the edit (with `scripts/run_answer_eval.py --input <subset>` or by patching the existing JSON in-place and re-running) would confirm.
2. **`read` stage is now the dominant cost** at ~38s avg / 50s p95. Two cheap levers worth measuring:
   - Reduce `ASK_READ_MAX_DOCS` from 3 → 2 (we already feed ~20k tokens; 2 docs roughly halves it).
   - Reduce `ASK_CHUNKS_PER_DOC` from 4 → 3.
   Both are pure-config changes. Worth a one-off eval re-run to see if quality holds.
3. **The cifar-autoresearch project shares the chat server.** With `--parallel 1` on the chat server, our requests queue behind theirs. Currently cifar is not running, but the next time both are active, our pipeline blocks. Options: coordinate timing, run a dedicated DOGV chat server on a different port, or move chat to `--parallel 2` (more VRAM cost but interleaved processing).
4. **`RERANK_TIMEOUT` bumped from 45s → 120s** because Qwen's rerank with reasoning exceeded the old cap and silently fell back. With `enable_thinking=False`, rerank takes 3-5s — the 120s cap is now pure headroom. Consider tightening back to 30-45s once we're confident, but it costs nothing to leave as is.
5. **`scripts/demo_smoke.py` not yet re-run on the no-think stack.** The original happy-path uses a deliberately no-match prompt that was the worst-case path under reasoning. Quick re-run after the eval-set fix would close out smoke.
6. **Phase F (ship polish) is still open**: Chainlit welcome copy refresh, systemd/cron unit for `scripts/maintain_indices.py --daily`, runbook polish. Not part of this session.
7. **Tune the validator-triggered fallback rate (currently 20%)**. Six cases triggered the concise-summary fallback after repair failed. The semantic claim guard (`answer_claim_guard_mode = unit_aware_strict`) is strict by design but may be over-flagging on Qwen. Could try `current_strict` mode (the previous default) and compare.
